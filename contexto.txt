# PROYECTO: PUMP.FUN INTELLIGENCE SYSTEM — CONTEXTO COMPLETO PARA CONTINUACIÓN

## DESCRIPCIÓN GENERAL

Sistema de ML para predecir qué coins lanzadas en pump.fun harán 2.5x sostenido en 4 horas, basado en el análisis de early buyers (primeros 3 minutos) y scoring histórico de wallets. El output principal es un score 0-1 por coin representando probabilidad de éxito, más un múltiplo esperado como output secundario.

## DEFINICIONES EXACTAS

### Early buyer
Wallet que realiza una compra en los primeros **180 segundos** desde la creación de la coin.
- Tier 1: first_entry_seconds ≤ 20
- Tier 2: first_entry_seconds > 20 y ≤ 180

### Creación de la coin
MIN(block_time) de todas las transacciones de compra de esa coin en dex_solana.trades con project='pumpdotfun' y token_sold_mint_address='So11111111111111111111111111111111111111112'. Es el primer comprador, no el deployer del contrato (limitación técnica con Dune).

### Label de éxito (label = 1)
Condición dual:
1. price_max_4h / price_at_5min ≥ 2.5
2. sustained_10min = 1 (existe al menos un bucket de 10 minutos entre t=5min y t=4h donde el precio mínimo observado ≥ 1.5 × price_at_5min)

label = 0 si no se cumple. label = NULL si price_at_5min es NULL o 0 (sin datos suficientes → excluida del entrenamiento).

### price_at_5min
Mediana (APPROX_PERCENTILE 0.5) de price_usd entre segundos 240 y 360 desde el lanzamiento. price_usd = amount_usd / token_bought_amount por transacción.

### Rug detection
rug_detected = TRUE si price_min_5min / price_max_5min < 0.2 (caída >80% en primeros 5 minutos).

### label_raw
Versión sin condición de sostenibilidad: 1 si price_max_4h / price_at_5min ≥ 2.5. Incluido para análisis comparativo, no usado en entrenamiento.

## PIPELINE DE DATOS DUNE

### Query 1: pumpfun_coins_by_week (ID: 6917014)
Parámetros: block_month, start_date, end_date. Paginación semanal. Extrae coin_address, created_at, creator_wallet.

### Query 2: pumpfun_early_trades_by_day (ID: 6917262)
Parámetros: block_month, start_date, end_date, end_date_extended (end_date + 2 días). Paginación diaria. Output **agregado por (coin_address, wallet_address)**: first_entry_seconds, total_sol_spent, total_usd_spent, n_trades, tier. Sin ORDER BY final (optimización de créditos). Sin tx_id ni block_time individuales. Reducción estimada 75-80% de volumen vs filas individuales. Coste estimado ~160-200 créditos/día (vs ~780 sin agregación).

Optimización clave: un único scan de dex_solana.trades usando MIN(block_time) OVER (PARTITION BY token_bought_mint_address) en lugar de dos CTEs separadas.

### Query 3: pumpfun_prices_by_day (ID: 6917275)
Parámetros: block_month, start_date, end_date, end_date_extended (end_date + 2 días para cubrir ventana 4h). Lógica de buckets de 10 minutos para sostenibilidad. Output: coin_address, price_at_5min, price_max_4h, price_min_5min, price_max_5min, label, label_raw, max_multiple, rug_detected, sustained_10min. Solo incluye coins con actividad entre minuto 4 y 6 (INNER JOIN en base_stats con price_at_5min_calc). Aproximadamente 2.000 coins/día de las ~23.000 lanzadas (el 91% muere antes del minuto 4, decisión aceptada: estas coins no son útiles para el modelo).

Lógica: consulta coin_prices JOIN coins WHERE label IS NOT NULL, ordena por created_at ASC, split en índice int(len × 0.8). Devuelve sets de coin_address. TRAIN_RATIO = 0.80.

## MÓDULO WALLET SCORING (src/wallet_scoring.py)

### Parámetros actuales
```python
ROI_CAP         = 10.0   # cap por coin antes de calcular media
MIN_APPEARANCES = 5      # mínimo para score_reliable = TRUE
PRIOR_STRENGTH  = 5      # pseudo-observaciones Bayesian
MAX_LIFT        = 8.0    # lift en que win_component satura (≈40% win rate con baseline 5%)
PENALIZATION    = 0.20   # peso penalización por rugs
```

### Lógica de cálculo
**Solo usa datos de train_coins** para evitar data leakage.

Carga: early_buyers JOIN coin_prices JOIN coins WHERE label IS NOT NULL AND coin_address IN train_coins.

Métricas por wallet:
- appearances_total: count de coins
- wins: sum(label)
- win_rate: wins / appearances_total
- avg_roi_capped: mean(min(max_multiple, ROI_CAP)), fillna=1.0
- negative_rate: mean(rug_detected)
- avg_entry_seconds: mean(first_entry_seconds), fillna=90.0
- pct_tier1_entries: mean(tier == 1)
- conviction_ratio: avg_sol en winners / avg_sol en losers, clip(0, 10). Si sin losers → 1.0. Si sin winners → 0.5.

**Bayesian win rate**: bayesian_wr = (wins + PRIOR × global_wr) / (appearances_total + PRIOR)
**Lift**: bayesian_wr / global_wr

### Fórmula del score
```python
win_component        = clip(lift / MAX_LIFT, 0, 1)          # peso 0.65
roi_component        = clip(avg_roi_capped / 5.0, 0, 1)     # peso 0.20
entry_component      = clip(1 - avg_entry_seconds/180, 0,1) # peso 0.05
conviction_component = clip((conviction_ratio-1)/3, 0, 1)   # peso 0.10
penalization         = negative_rate × PENALIZATION

performance_score = clip(
    win_component*0.65 + roi_component*0.20 +
    entry_component*0.05 + conviction_component*0.10 - penalization,
    0, 1
)
```

score_reliable = TRUE si appearances_total ≥ 5. Wallets sin score_reliable reciben performance_score calculado igual (no se fuerza 0.1 — el Bayesian shrinkage estabiliza automáticamente).

### Estado actual (última ejecución)
- 320,744 wallets totales
- 67,504 wallets fiables (appearances ≥ 5)
- 184 wallets fiables con score ≥ 0.5
- Win rate global (train): ~6.35%
- Rug rate global (train): ~42.34%
- Score medio wallets fiables: ~0.105

## MÓDULO FEATURES (src/features.py)

### Anti-leakage en co-ocurrencia
Paso 1: Construir pair_counts SOLO con coins exitosas de train_coins.
Paso 2: Aplicar lookup de pair_counts a TODAS las coins (train + test) → no hay leakage porque los pares se calculan solo desde train.

MAX_WALLETS = 25 por coin (top por total_sol_spent) para limitar RAM en self-join.
BATCH_SIZE = 3,000 coins por lote.

### Features calculadas

**Bloque 1 — Composición de wallets** (groupby coin_address sobre eb con wallet_metrics ya mergeado via SQL):
- n_early_buyers: nunique(wallet_address)
- n_reliable_wallets: sum(score_reliable)
- avg_wallet_score: mean(performance_score)
- max_wallet_score: max(performance_score)
- pct_high_score_wallets: mean(performance_score > 0.6)
- pct_negative_wallets: mean(negative_rate > 0.4)
- pct_new_wallets: mean(appearances_total < 3)

**Bloque 2 — Co-ocurrencia**:
- avg_cooccurrence_score: media de cooc_count entre todos los pares de wallets de la coin, donde cooc_count = número de coins exitosas de train en que ese par coincidió.

**Bloque 3 — Comportamiento de compra**:
- total_volume_sol: sum(total_sol_spent)
- avg_buy_size_sol: mean(total_sol_spent)
- std_buy_size_sol: std(total_sol_spent), fillna=0
- concentration_top5: suma SOL top 5 wallets / total_volume_sol
- n_tier1_buyers: sum(tier == 1)
- creator_is_buyer (Opción B implementada): TRUE si la wallet con wallet_address == creator_wallet realizó n_trades > 1 en los primeros 3 minutos. **Nota**: creator_wallet en coins es el primer comprador (limitación Dune), no el deployer del contrato. Con Opción A (forzar FALSE) se descartó porque el 100% salía TRUE. Con Opción B actual: ~5,340 coins con creator_is_buyer=TRUE de 52,218.

**Bloque 4 — Velocidad de acumulación**:
- buys_in_first_20s: sum(first_entry_seconds ≤ 20)
- buys_20s_to_60s: sum(20 < first_entry_seconds ≤ 60)
- buys_60s_to_180s: sum(60 < first_entry_seconds ≤ 180)
- acceleration_ratio: buys_first_90s / max(buys_last_90s, 1)
- time_to_5th_buy: 5th percentile of first_entry_seconds (NaN si < 5 compradores)

**Bloque 5 — Contexto temporal** (desde coins.created_at):
- hour_utc: hora UTC (0-23)
- day_of_week: 0=lunes, 6=domingo

## DECISIONES TÉCNICAS CLAVE Y SU RAZONAMIENTO

### Decaimiento exponencial en wallet scoring: DESCARTADO
Se discutió pero se descartó en la versión actual. El scoring usa historial completo sin decaimiento. Motivo: insuficientes datos para que las ventanas temporales sean significativas. Previsto para implementación futura cuando haya 60-90 días de datos.

### Concentración top3 → top5: IMPLEMENTADO
Cambio menor decidido por el usuario. Se usa concentration_top5.

### Ventana early buyers: 180 segundos (3 minutos)
Decisión fija. Tier 1 = primeros 20s, Tier 2 = 20-180s. No hay Tier 0 separado en la implementación actual (originalmente se discutió pero se integró en Tier 1/2).

### creator_is_buyer: Opción B activa
Opción A (forzar FALSE) implementada inicialmente porque 100% salía TRUE. Luego se implementó Opción B: TRUE si el primer comprador (creator_wallet) hizo n_trades > 1 en primeros 3 minutos. Limitación conocida: creator_wallet en Dune es el primer comprador, no el deployer real del contrato.

### Split temporal 80/20 compartido
Todos los módulos usan splitter.get_train_test_coins(). No hay fechas hardcodeadas. Se adapta automáticamente al volumen de datos disponible.

### Query de prices: INNER JOIN (no LEFT JOIN) deliberado
Se detectó que LEFT JOIN devolvía ~23k coins/día pero la mayoría sin label útil. Con INNER JOIN sobre coins con actividad entre minuto 4-6, se obtienen ~2k coins/día con label calculable. **Decisión aceptada**: el 91% de coins sin precio a 5min no son útiles para el modelo y no se incluyen.

### Dune como única fuente histórica (no BigQuery, no Bitquery)
BigQuery: dataset público de Solana desactualizado, queries con errores estructurales, coste prohibitivo en free tier. Bitquery: ofrecieron precio elevado al contactarles. Helius: reservado para tiempo real.

### No usar ORDER BY en query de trades
Eliminado para reducir coste en créditos Dune. Pandas reordena localmente si es necesario.

### Datos en CSVs individuales por día/semana
Scripts reanudables: si existe el archivo, se salta. Permite interrupción y reanudación sin reprocesar.

### Filtro de sostenibilidad (buckets 10 min)
Implementado en SQL directamente en Dune (Query 3). Alternativa de calcular en Python descartada porque requería serie temporal completa no disponible en el output actual.

## PROBLEMAS IDENTIFICADOS Y ESTADO

### Data leakage: RESUELTO
Versión anterior de features.py calculaba co-ocurrencia con todas las coins incluyendo test. AUC-ROC bajó de 0.974 → 0.667 tras la corrección, confirmando que el leakage estaba inflando métricas artificialmente. wallet_scoring.py también corregido para usar solo train_coins.

### Wallet scores casi todos en 0.1: PARCIALMENTE RESUELTO
Causa: con baseline ~5-6%, el lift de wallets mediocres es bajo. Solución implementada: Bayesian shrinkage + lift relativo + conviction ratio + entry speed. Problema residual: insuficientes datos para que la distribución de scores sea discriminativa. Con más días mejorará.

### Regresor sobreestima (pred_mean 2.20x vs target_mean 1.41x): CONOCIDO
No resuelto. El regresor predice expected_multiple para coins con max_multiple NOT NULL, pero hay sesgo de selección porque solo las que llegaron a tener precio calculado están en el dataset. No es prioritario actualmente.

### avg_cooccurrence_score domina si hay leakage: RESUELTO
Tras corrección anti-leakage, bajó de 57% a 11.75% de importancia, que es razonable.

### creator_is_buyer: 100% TRUE con Opción A: RESUELTO con Opción B
Ver sección anterior. Limitación de Dune (creator_wallet = primer comprador).

## CONTEXTO ADICIONAL RELEVANTE

### Escala de pump.fun
~23,000 coins lanzadas por día. Solo ~9% tienen suficiente actividad para tener price_at_5min calculable. De esas, ~5% son label=1.

### Desbalanceo de clases
~95% label=0, ~5% label=1. Manejado con scale_pos_weight automático en XGBoost. Métrica principal: AUC-PR (no AUC-ROC) por ser más informativa con desbalanceo severo.

### Ventana de precio para label
- Referencia: mediana de precios entre minutos 4-6 (seconds 240-360)
- Máximo: entre minuto 5 y hora 4 (seconds 300-14400)
- Rug: comparación min/max en primeros 5 minutos
- Buckets sostenibilidad: ventanas de 600s entre seconds 300-14400

### Limitación del regresor
Entrena solo sobre coins con max_multiple NOT NULL (las que tienen precio calculado, no solo label=1). Target clippeado a 50x para evitar outliers extremos.

### Tamaño actual del modelo
classifier: 31 árboles (early stopping a las 11 iteraciones sobre 300 estimadores). regressor: 26 árboles.

### Estado del README.md
Contiene un backlog informal con ideas de mejoras futuras (features de smart money, co-ocurrencia avanzada, embeddings de wallets, graph features). Sirve como referencia de roadmap pero nada implementado de lo listado ahí salvo lo ya descrito en este documento.
